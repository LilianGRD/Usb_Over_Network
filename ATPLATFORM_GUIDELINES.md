## Architecture Principles
- Each person should have their own atSign (e.g. @alice, @bob) — this is the core identity unit.
- Agents and applications can either have a dedicated atSign OR share an atSign with their owner, using namespaces for isolation. Multiple agents/apps on one atSign use different namespaces to keep their data separate.
- Multiple instances of the same agent can run under a single atSign (see Multi-Agent Coordination below).
- IoT devices can have their own atSign for autonomous operation, or be managed as keys under a person's atSign.
- All data is end-to-end encrypted by default. Only the atSign owner holds the decryption keys.
- There is NO application backend. atServers act as personal data stores and rendezvous points.
- Use namespaces to separate application data (e.g. "myapp" becomes key.myapp@atsign).
- Use AtKey with sharedWith to share encrypted data with specific atSigns.
- Use AtKey with metadata.isPublic = true only for intentionally public data.
- Do NOT store application data in local files. Store data as AtKeys so it automatically synchronizes across all of the atSign owner's devices. Local files are invisible to other devices; AtKeys flow via sync.

## Initialization Pattern (Dart CLI / Agents)
- Use AtClientManager.getInstance().setCurrentAtSign(atSign, namespace, preferences) to initialize.
- Set AtClientPreference with rootDomain: AtRootDomain.atsignDomain, hiveStoragePath, commitLogPath, and namespace.
- IMPORTANT: When running multiple agents or agent instances on the same machine, each MUST use a unique hiveStoragePath and commitLogPath (e.g. a temp directory with a UUID: Directory.systemTemp.createTempSync("agent_").path). If two processes share the same hive path, the hive boxes will collide and throw errors.
- For agent/CLI authentication use at_cli_commons (CLIBase): AtClient atClient = (await CLIBase.fromCommandLineArgs(args)).atClient; — handles arg parsing, atKeys loading, and PKAM auth in one line.
- APKAM enrollment for new devices is supported via EnrollmentService in at_client.

- **Switch atSign**: Let user pick a different atSign from the keychain and re-authenticate (KeychainStorage().getAllAtsigns() → select → re-auth).

## AtKey Construction
- Cascade: AtKey()..key = "name"..namespace = "app"..sharedWith = "@bob" — matches official examples.
- Typed builders (preferred for type safety): AtKey.self("name"), AtKey.shared("name", namespace: "app"), AtKey.public("name"), AtKey.local("name", "@alice").
- AtKey.fromString("@bob:name.app@alice") to parse a full atProtocol key string.
- All key names are forced lowercase. Values remain case-sensitive.

## Data Operations
IMPORTANT: Always store application data as AtKeys rather than writing to local files. AtKeys automatically synchronize across all of the atSign owner's devices over time. Local files stay on a single machine and break the atPlatform's multi-device data flow model.
- put(AtKey, value, {PutRequestOptions? putRequestOptions}): Store/share data. Use sharedWith for encrypted sharing with another atSign.
  IMPORTANT: To send data directly to the remote (cloud) secondary server, pass putRequestOptions: PutRequestOptions()..useRemoteAtServer = true. Without this, put() writes only to the local secondary and waits for sync.
- get(AtKey, {GetRequestOptions? getRequestOptions}): Retrieve data. Returns AtValue with .value property.
  IMPORTANT: To read from the remote secondary rather than the local cache, pass getRequestOptions: GetRequestOptions()..useRemoteAtServer = true. Use ..bypassCache = true to skip cached data from other atSigns.
- delete(AtKey, {DeleteRequestOptions? deleteRequestOptions}): Remove data. Pass DeleteRequestOptions()..useRemoteAtServer = true to delete directly from the cloud secondary.
- uploadFile / downloadFile / shareFiles: Built-in file transfer methods on AtClient — no separate package needed.
- notificationService.notify(NotificationParams.forUpdate(key, value: data)): Push data to another atSign in near real-time. NOTE: NotificationParams.forText() is deprecated — use forUpdate() instead.
- notificationService.subscribe(regex: pattern, shouldDecrypt: true): Listen for incoming notifications.
- KeyStream / ListKeyStream / MapKeyStream / SetKeyStream: Reactive stream-based key watching — subscribe to live changes on specific keys or key patterns.
- For RPC patterns use the built-in AtRpc / AtRpcClient classes (production-ready since 3.9.1, not experimental).

## Multi-Agent Coordination
When building agents that may run as multiple instances (for load balancing or redundancy), two proven patterns exist:
### Pattern 1: Immutable Mutex Race
- Create an AtKey with Metadata()..immutable = true and a short ..ttl (e.g. 60000ms).
- Each agent instance tries to put() this key with useRemoteAtServer = true when a task arrives.
- Only the first put() succeeds (the immutable flag causes subsequent puts to throw). The winner processes the task; losers skip it.
- This gives automatic distributed locking with no external coordination service. The TTL auto-cleans the mutex key.
- Reference: https://github.com/cconstab/personalagent/blob/main/agent/lib/services/at_platform_service.dart
### Pattern 2: Ephemeral Stateless Instances (NoOp Sync)
- Use ServiceFactoryWithNoOpSyncService() when creating the AtClient — this disables the sync service entirely.
- Use per-session ephemeral storage — each instance MUST have a unique hiveStoragePath and commitLogPath (e.g. Directory.systemTemp.createTempSync("agent_").path). If two instances share the same hive path on the same machine, the hive boxes will collide and throw errors.
- Each instance subscribes to notifications and handles sessions independently, routed by unique session IDs.
- No mutex needed because work is partitioned by session/notification routing, not raced for.
- This is the pattern used by srvd, sshnpd, and npp in NoPorts.
- Reference: https://github.com/atsign-foundation/noports/blob/trunk/packages/dart/sshnoports/bin/srvd.dart

## Networking, File Transfer & Access Control
These patterns extend the atPlatform beyond data sharing into TCP connectivity, bulk file movement, and policy enforcement:

### E2E Encrypted TCP Tunnels (NoPorts)
Use noports_core to create E2E encrypted TCP connections with zero open inbound ports on either side.
A local listening socket tunnels traffic through the atPlatform to a remote device — no firewall changes needed.
- Pub package: noports_core. Key class: Npt. CLI: npt binary.
- Requires a socket rendezvous (srvd) atSign and a daemon on the remote device.
- Docs: https://docs.noports.com/ and https://pub.dev/packages/noports_core

### Encrypted File Transfer
For bulk file transfer: encrypt the file client-side, upload the encrypted blob to a web file service, then send the URL and encryption key to the recipient as an encrypted AtKey. The recipient downloads the blob and decrypts locally.
- For simpler cases use atClient.uploadFile() / downloadFile() / shareFiles() which handle encryption automatically.
- Store config (e.g. storage server URL) as AtKeys — never local files.
- Reference implementation: https://github.com/cconstab/furl

### Policy Enforcement (NPA/NPP)
For access control: run a policy service on a dedicated atSign that receives authorization requests via AtRpc and returns allow/deny decisions.
- Policy data is stored as encrypted AtKeys — never local files. Syncs across devices automatically.
- The pattern is reusable for any authorization logic. See NoPorts NPA/NPP for a working implementation.
- Reference: https://github.com/atsign-foundation/noports (npp_atserver.dart, npp.dart)

- Use meaningful key names: e.g. "temperature.iot" not "data123".
